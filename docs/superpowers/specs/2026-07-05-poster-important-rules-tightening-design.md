# Demo 改造设计：海报无字约束与路径顺序约束收紧

## 1. 目标

* 基于当前 `route-driven` 架构，收紧海报生成中“图中有字”和“路径顺序”两类高频问题。
* 明确 `P2`、`P3`、`route.md` 契约三层职责，避免继续把全局布局问题误塞进 event 级 `subject/avoid`。
* 把当前散落在 `route-markdown.ts` 内部的 `Important Rules` 提炼为共享规则源，但不破坏 `route.md` 作为权威输入的现有架构。

## 2. 非目标

* 本轮不引入程序化路径布局算法，不做几何强控渲染。
* 本轮不引入 OCR、图像审核器、negative prompt、seed 固定或额外图像后处理链路。
* 本轮不改动 `route.md` 的 parser schema 结构；只调整规则来源与规则内容。
* 本轮不把点位名称、编号改为程序后绘制；仍由 Seedream 在整图中生成。

## 3. 已确认决策

* 采用方案 B：新增共享规则源模块，命名固定为 `src/engine/prompts/p3-poster-important-rules.ts`。
* `route.md` 中的 `## Important Rules` 仍然保留，且继续作为 `P3/P4` 的权威输入之一。
* `route-markdown.ts` 不再持有规则字面量本身，只负责把共享规则写入 `route.md`。
* “图中有字”属于整图级视觉约束，主修复层在 `P3` 顶部重要事项区，不以 `P2 subject` 反例为主。
* “路径顺序”属于整图级构图约束，主修复层在 `P3` 和共享规则源，不由 `P2` 承担。
* `P2` 只做 event 局部语义收紧：让 `subject` 更具体，让 `avoid` 更聚焦文字噪声和误导元素。

## 4. 当前代码现状

### 4.1 `Important Rules` 的定义位置不合理，但其输出位置合理

* `src/engine/renderers/route-markdown.ts` 当前在文件内部直接定义 `IMPORTANT_RULES`。
* 同文件会把这些规则写入 `## Important Rules` 区块。
* 由于 `src/engine/parsers/route-markdown.ts` 会解析该区块，`route.md` 已经是 `P3/P4` 的真实权威输入之一。

结论：
当前问题不是“renderer 不该输出 Important Rules”，而是“renderer 不该拥有规则定义本身”。

### 4.2 `P3` 已有顺序约束，但强度不够

* `src/engine/prompts/p3-poster.ts` 当前已声明“路线节点必须按给定顺序连续编号，禁止重排、跳号、合并或交换顺序”。
* 但现有文案仍偏语义约束，没有把“主阅读方向必须从左到右”“节点空间位置需与编号一致”写成更几何化的硬约束。
* 当前也没有显式声明“名称和序号只能在节点外部，禁止嵌进主体图标内部”。

### 4.3 `P2` 目前承担了部分本不属于它的纠偏期待

* `src/engine/prompts/p2-event-visual-brief.ts` 当前主要控制 `subject/avoid` 输出格式和局部语义质量。
* 用户已验证：仅在 `P2 subject` 中增加反例，不能稳定解决“图中有字”。
* 根因在于“图中有字”大多发生在整图渲染阶段，而非 event 语义补全过程。

## 5. 问题拆分

### 5.1 “图中有字”

问题表现：
* event 图标内部混入菜单字样、招牌字样、屏幕字幕、海报字样、logo 或数字。
* 某些抽象场景词会诱发模型生成带字的牌面、灯牌或屏幕。

问题层级：
* 一级问题：整图输出约束不足，属于 `P3`。
* 二级问题：event 语义过抽象，属于 `P2`。

### 5.2 “路径顺序”

问题表现：
* 模型容易把路线理解成“自上而下”的阅读流。
* 左右横向贯穿式地图中，节点编号与视觉空间顺序可能脱钩。

问题层级：
* 完全属于整图构图约束，应在共享规则源 + `P3` 中处理。
* `P2 subject/avoid` 不应承担路径走向控制。

## 6. 方案选型

### 6.1 方案 A：仅在 `route-markdown.ts` 与 `p3-poster.ts` 内补文案

优点：
* 改动最小。

缺点：
* 规则定义继续散落，职责边界依旧含混。
* 后续若 `P3` 与 route 输出规则不同步，容易再次漂移。

### 6.2 方案 B：抽共享规则源 `p3-poster-important-rules.ts`

优点：
* 规则定义集中，`renderer` 与 `prompt` 职责清晰。
* 仍保持 `route.md -> parser -> p3` 的单一路径，不破坏现有架构。
* 后续新增全局生图硬规则时，维护点单一。

缺点：
* 需要补一层模块引用与测试同步。

### 6.3 方案 C：在 B 基础上增加程序化布局

优点：
* 对路径顺序最稳定。

缺点：
* 范围明显超出本轮“提示词与契约收紧”。

结论：
采用方案 B。

## 7. 推荐方案设计

### 7.1 新增共享规则源模块

新增文件：
* `src/engine/prompts/p3-poster-important-rules.ts`

职责：
* 持有全局海报硬规则的唯一字面量来源。
* 输出给 route renderer 使用，写入 `route.md`。
* 不直接让 `p3-poster.ts` 再次硬编码同一批规则，避免双写。

### 7.2 `route-markdown.ts` 的职责调整

改动原则：
* 保留 `## Important Rules` 输出。
* 删除文件内部本地 `IMPORTANT_RULES` 常量。
* 改为从 `p3-poster-important-rules.ts` 导入共享规则。

这样处理后：
* `route.md` 仍是权威输入。
* renderer 只是规则的“落盘者”，不再是规则的“拥有者”。

### 7.3 共享规则内容调整

在共享规则源中补入以下两类硬约束：

1. 无字约束
* 所有 event 主体图标内部禁止出现任何可读文字、字母、数字、logo、水印、菜单、招牌、屏幕字幕。
* 点位名称和编号只能作为节点外部标注，不能嵌入主体图标内部。
* 若场景天然包含店招、路牌、菜单、电子屏，统一简化为无字形状，不保留可读字符。

2. 路径顺序约束
* 主路径必须从画面左侧起、向右侧结束。
* 节点编号顺序必须与空间阅读顺序一致。
* 节点中心点横向位置应随编号递增，允许轻微上下起伏，但不允许以上下纵向堆叠作为主顺序。
* 不允许回头路、环形主路径或让装饰元素制造错误阅读顺序。

### 7.4 `p3-poster.ts` 的调整方式

`p3-poster.ts` 继续从 `params.route.importantRules` 读取 route 中的规则，不直接重新声明整批规则。

本轮在 `P3` 中追加的内容只保留“解释性补强”，不做重复定义：
* 把“名称约束”和“节点外部标注”衔接得更明确。
* 把“左到右主阅读轴”用更几何化措辞表达。
* 把“不要复用参考图中的地图布局”改成更精确说法，避免误伤“可学习横向阅读节奏”这一目标。

说明：
当前 `P3` 里“不得复用参考图中的地图布局”是合理边界，但文案需要避免把“版式节奏”与“杭州特定布局”混为一谈。

### 7.5 `P2` 的调整边界

`p2-event-visual-brief.ts` 仅做以下收紧：

* `subject` 示例与反例更偏具体器物、食物、动作、真实空间。
* 明确弱化“幻境、沉浸式、流光溢彩”这类高抽象词。
* `avoid` 增加更高价值示例：`招牌文字`、`菜单字样`、`屏幕字幕`、`价格海报`。

不做的事：
* 不把“左到右路径”“节点外部标注”等全局规则塞进 `subject` 或 `avoid`。

## 9. 验收与测试

### 9.1 契约验收

* 新生成的 `route.md` 仍包含 `## Important Rules`。
* `Important Rules` 内容来自共享规则源，而不是 renderer 内嵌字面量。
* parser 无需改 schema 即可继续读取 route。

### 9.2 Prompt 验收

* `P3` prompt 中明确出现“图标内部不得有字”“名称和序号只能在节点外部”“主路径左起右终”“节点 x 坐标随编号递增”这类约束。
* `P2` prompt 中新增针对文字噪声的高价值 `avoid` 示例，但不新增全图布局类约束。

### 9.3 测试建议

* `route-markdown.test.ts`
  * 断言 `route.md` 中输出新的共享规则内容。
* `poster-prompt.test.ts`
  * 断言 prompt 中包含“节点外部标注”“图标内无字”“左到右主路径”相关文案。
* `event-visual-brief.test.ts`
  * 断言 `avoid` 示例覆盖文字噪声。
  * 断言 `subject` 反例继续压制高抽象场景词。

## 10. 风险与后续

### 10.1 本轮已知风险

* 纯提示词方案只能显著收敛问题，不能把路径顺序做成 100% 几何确定。
* 若风格参考图本身含强文字元素或强纵向流向，仍可能对模型造成反向拉扯。
* `P3` 文案过长会带来规则互相稀释风险，因此本轮应优先保留少量强约束，避免同义反复堆叠。

### 10.2 后续观察项

若本轮上线后，“路径顺序”仍不稳定，则下一轮再单独立项评估：
* 程序化主路径与节点文字覆盖层。
* 风格参考图重选或新增“横向阅读更稳定”的参考图资产。
* 对生成结果做轻量 OCR/版式验签。
