# Write Refactor Spec

用于在现有项目中编写“改造类 / 对齐类 / 重构类”Spec。

## 触发形式

在 Trae 中调用本 Skill，默认有两种触发形式：

### 1. 基于多轮对话共识写 Spec

适用情况：

- 用户和 Trae 已经经过多轮讨论
- 一系列修改点、边界和风险已经基本达成共识
- 用户调用本 Skill，是为了把这些共识结构化落到 Spec 中

此时本 Skill 的重点是：

- 提炼已确认决策
- 补齐结构
- 收口范围
- 把阶段门禁、验收标准和风险隔离写清

### 2. 基于已有修改文档产出规范化副本 Spec

适用情况：

- 用户已经通过其他方式拿到一份修改文档、方案文档或需求草稿
- 这份文档不符合本 Skill 的标准结构
- 或文档缺失阶段门禁、验收标准、Phase 拆分等关键内容
- 用户调用本 Skill，并明确指向该文档

此时本 Skill 的重点是：

- 在已有文档基础上提取有效内容
- 重新组织为本 Skill 的标准 Spec 结构
- 补齐缺失的门禁、Phase 模板、风险和验收口径
- 产出一份符合规范的副本 Spec，而不是要求用户从头重写

默认规则：

- 除非用户明确要求覆盖原文，否则不直接改写原始文档
- 默认输出为一份新的副本 Spec
- 如果原文已有清晰结论，优先复用，不擅自改动其产品结论
- 仅对结构、门禁、验收、范围收口进行规范化补全
- 若原文存在关键缺口且无法合理推断，再向用户提问

适用场景：

- 现有模块需要重构、拆分、收口范围
- 体验优化涉及多页面、多接口、多阶段实施
- 用户明确要求先写 Spec / 设计文档 / 分阶段计划
- 需要把聊天里的决策、非目标、风险和阶段门禁固化成文档
- 用户要求把一份已有修改文档整理成符合本 Skill 规范的副本 Spec

不适用场景：

- 单一小 Bug 修复，且不需要设计文档
- 从零搭建新产品或全新页面
- 已经有完整 Spec，本轮只需要执行代码

## 核心目标

产出一个可执行、可验收、可分阶段推进的 Refactor Spec，避免以下问题：

- 愿望清单化
- 范围膨胀
- 方案、实现、风险混写
- 阶段边界不清
- 执行过程跨阶段连续改代码

本 Skill 只负责写或整理 Spec，不负责直接开始实现。

## 工作方式

1. 先判断当前属于哪种触发形式：
   - 多轮对话共识落 Spec
   - 已有修改文档规范化为副本 Spec
2. 读取与任务相关的现有 Spec、代码、最近决策、约束文档或用户指向的源文档。
3. 若存在关键模糊点，只问会影响结构或范围判断的问题。
4. 先收敛目标、非目标和阶段边界，再写实现思路。
5. 若用户提供参考 Spec，优先复用其抽象层级、措辞口径和版式节奏。
6. 默认将文档写入 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`；若用户指定路径，以用户为准。

## 基于已有文档时的处理规则

当用户是通过“指向已有修改文档”的方式调用本 Skill 时，按以下规则执行：

1. 先抽取原文中已经存在的：
   - 目标
   - 非目标
   - 已确认决策
   - 范围项
   - 风险项
   - 已有分阶段信息
2. 再识别原文缺口，重点检查：
   - 是否缺少标准结构
   - 是否缺少阶段执行门禁
   - 是否缺少 Phase 模板中的“建议改动面 / 验收标准 / 风险提示”
   - 是否把方案、范围、实现细节混写
3. 产出一份规范化副本 Spec，默认不覆盖原文。
4. 对原文未明确的内容，只做结构化补全，不擅自发明产品决策。
5. 若必须新增决策才能成文，先向用户确认。

## 提问规则

如果信息不全，可以向用户提问，但要遵守以下约束：

- 只问真正影响 Spec 结构或范围判断的问题
- 一次优先 1 个问题，最多不超过 3 个
- 优先澄清：
  - 本轮目标
  - 非目标
  - 是否需要分阶段
  - 是否需要硬门禁
  - 高风险项是否必须单独成 Phase

## 固定结构

除非用户明确要求调整顺序，否则 Spec 按以下顺序输出：

1. `目标`
2. `非目标`
3. `已确认决策`
4. `当前代码现状`
5. `本轮范围总表`
6. `分阶段计划`
7. `阶段执行门禁`
8. `总体验收顺序`
9. `方案选型`
10. `目标设计`

## 各章节写法

### 1. 目标

- 只写本轮要解决什么
- 说明为什么需要收口，避免愿望清单化

### 2. 非目标

- 明确本轮不处理什么
- 列出容易被顺手带上的邻近需求

### 3. 已确认决策

- 把聊天里已拍板的结论固化成列表
- 已确认内容不要再写成“待讨论”

### 4. 当前代码现状

- 按“文件位置 + 当前行为 + 缺口”来写
- 尽量定位到真实路径、接口、组件或状态结构

### 5. 本轮范围总表

- 使用 `ID / 优化项 / 目标状态 / 高层方案`
- 每项都要能映射到后文某个 Phase 或目标设计小节

### 6. 分阶段计划

- 按 Phase 拆开写
- 低风险与高风险改动不要混写
- 每个 Phase 必须使用统一模板

### 7. 阶段执行门禁

- 把协作流程写成硬规则
- 至少覆盖：阶段开始前、阶段完成后、下一阶段开始前

### 8. 总体验收顺序

- 解释为什么先做 A 再做 B
- 顺序必须服务于风险控制，而不是只写时间顺序

### 9. 方案选型

- 列候选方案
- 明确选哪个、为什么、不选什么
- 写清成立边界，不夸大兼容性

### 10. 目标设计

- 按具体场景拆开写，不混写
- 同时覆盖页面行为、接口语义、状态字段、异常分支

## 固化的可复用逻辑：Phase 模板

以下 Phase 写法是本 Skill 的固定模板。除非用户明确要求修改，否则不要改形状：

```md
### Phase X：阶段名

**执行 Agent 可用 Skills**
* `...`

**范围**
* ...

**建议改动面**
* `path/to/file`

**验收标准**
* ...

**风险提示**
* ...
```

### Phase 模板使用规则

- `执行 Agent 可用 Skills`：只写本阶段真正需要的 Skill
- `范围`：写任务项，不写零散实现碎片
- `建议改动面`：尽量落到真实文件路径
- `验收标准`：必须写成可观察、可验证结果
- `风险提示`：明确说明为什么该阶段要隔离执行

## 推荐写法

以下不是硬模板，但默认采用：

- 先写硬约束，再写实现建议
- 先写当前差距，再写目标状态
- 每个优化项都要落到“可验收状态”
- 高风险改动单独成 Phase
- 多用“允许什么 / 禁止什么”来收口语义边界
- 架构方案要写清边界条件，避免把 Demo 方案写成通用基础设施方案
- 页面、接口、状态机、失败恢复尽量成组出现，避免只写 happy path

## 阶段执行门禁的默认写法

如果用户没有给出另一套流程，默认在 `阶段执行门禁` 中覆盖以下三段：

### 1. 阶段开始前

- refresh memory / 阅读最新约束
- 汇报本阶段目标、计划改动文件、验收方式、风险点
- 请求用户确认是否开始
- 得到批准后才允许编码

### 2. 阶段完成后

- 完成本阶段验证
- 运行 `git status` 与 `git diff --stat`
- 推荐 commit message
- 汇报结果并请求用户确认
- 未确认前不得直接进入下一阶段

### 3. 下一阶段开始前

- 再次 refresh memory
- 汇报下一阶段概览
- 请求确认
- 得到批准后再实施

## 输出要求

- 语言默认中文，除非用户另有要求
- 结论先行，分析在后
- 结构清晰，但不要为了凑格式写空话
- 每一节都要服务于“执行时可落地”

## 交付检查

完成前自检：

1. 是否已经把目标和非目标分开
2. 是否已经把已确认决策从“讨论态”改成“结论态”
3. `当前代码现状` 是否都带了真实文件或模块定位
4. Phase 是否都使用了统一模板
5. 是否已经写出明确验收标准和风险提示
6. 是否存在跨章节重复、冲突或范围漂移
7. 是否能直接给执行 Agent 用，而不用二次解释

## 示例触发语句

- “帮我写一个重构 Spec”
- “先别改代码，先把这个多阶段改造写成设计文档”
- “把这次 UX / 架构调整收敛成可执行 Spec”
- “给这个 refactor 写一个带 Phase 门禁的方案文档”
- “我们前面已经聊清楚了，用 write-refactor-spec 帮我落成 spec”
- “基于这个修改文档，帮我出一份符合 write-refactor-spec 规范的副本 spec”
