# 规则：高难度任务自动检测

## 执行流程

### 步骤 1：检查未提交更改
执行 `git status`。
如果有未提交的更改：
- 向用户展示当前未提交的文件列表
- 强烈建议先 commit，使用 auto-commit 规则生成 commit message
- 说明理由："高风险操作前应保存当前工作进度，以便出现问题时可以快速回滚"

### 步骤 2：评估是否需要新建分支
基于以下因素给出是/否建议：

**建议新建分支（是）的情况：**
- 任务预计修改超过 5 个文件
- 涉及核心业务逻辑变更
- 可能影响现有功能稳定性
- 需要长时间开发（超过 30 分钟）
- 涉及数据库迁移、API 变更等不可逆操作
- 需要测试验证后才能合并

**可以直接在主分支操作（否）的情况：**
- 仅修改单一文件且逻辑简单
- 纯样式/文案调整
- 明确的 bugfix 且改动范围极小
- 项目处于早期阶段，尚未发布

### 步骤 3：输出建议报告
向用户展示结构化评估：
```
⚠️ 检测到高风险任务

当前未提交更改：[数量] 个文件
├── 建议操作：先执行 commit / 无需 commit
│   └── 理由：...

是否需要新建分支：[是/否]
├── 理由：...
├── 建议分支名：<branch-name>
└── 建议工作流：commit -> checkout -b -> 开发 -> 测试 -> merge
```

## 强制保护规则
以下场景必须拒绝在主分支直接执行，必须新建分支：
1. 涉及删除或重命名正在使用的公共 API
2. 涉及数据库 schema 变更（如迁移文件）
3. 涉及修改 CI/CD 配置
4. 涉及升级核心依赖大版本（如 Next.js 13 -> 14）

## 通用准则
- 所有建议必须以用户确认后执行，不得擅自运行 git commit 或 git branch 命令
- 使用中文与用户沟通（用户偏好语言）
- 保持简洁，重点突出风险点和行动建议
- 如果检测到 .git 不存在，跳过所有 git 相关检查并提示用户初始化仓库
